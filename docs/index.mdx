---
title: What is Dashbored?
description: A calm, private front door for the services that run your home server.
---

Dashbored brings service health, practical controls, container discovery, and launch links into one local-first console. It is built for the person operating the infrastructure—not for a public status page or wall display.

## One operational view

Connect the services you already trust, then use the dashboard to see what needs attention before opening an individual application. Native integrations provide focused data; everything else can remain a lightweight Launcher entry. Dashbored is deliberately not a generic proxy: your browser never receives upstream credentials or authenticated API responses.

## The core pieces

| Piece | What it does |
| --- | --- |
| Dashboard | The authenticated web console. It reads the database and renders service pages, history, actions, and Launcher. |
| Worker | A separate, low-frequency process that checks configured services and synchronizes Portainer inventory. |
| PostgreSQL | Stores settings, encrypted credentials, compact health snapshots, session records, and action audit events. |
| Portainer connection | The v1 source of container inventory and the only provider able to perform container actions. |

## Before you begin

- Keep Dashbored and the services it can reach on a LAN, VPN, or private overlay such as Tailscale.
- Decide on an address that the dashboard container can resolve and reach—usually a private DNS name such as `service.home.arpa`.
- Use a unique password for the Dashbored owner account and unique, scoped credentials for each connected service.
- Run the worker continuously. A running web UI alone does not update health or Launcher data.

<Note>
Dashbored's URL validation only permits private LAN and Tailscale destinations. This prevents the dashboard from becoming an arbitrary request relay.
</Note>

## Recommended path

1. [Install Dashbored](/docs/install) with Docker Compose.
2. [Set production configuration](/docs/configuration), including the authentication URL and encryption key.
3. Sign in with the bootstrap owner account.
4. [Connect Portainer](/docs/portainer) and allow the worker to discover containers.
5. Use [Launcher](/docs/launcher) for every container that does not need a native dashboard.

## Supported v1 integration

- Portainer for discovery and confirmed container actions.
- Generic Launcher entries for private HTTP(S) applications.

Other adapters may appear in the interface while they mature. Treat Portainer as the supported v1 integration boundary and check [Service connections](/docs/services) for the current behavior of each adapter.
