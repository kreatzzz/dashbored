---
title: Authentication and owner access
description: Sign in safely, use a stable origin, and understand first-run account creation.
---

Dashbored is an owner-operated console. Password sign-in is enabled, while self-service sign-up is disabled after the first-run migration completes.

## First sign-in

The `migrate` service creates the bootstrap owner only if no users exist in PostgreSQL. Sign in with `BOOTSTRAP_EMAIL` and `BOOTSTRAP_PASSWORD` from the deployment environment file.

If the first account was not created, check migration logs:

```bash
docker compose logs --tail=150 migrate
```

The password must be at least 12 characters. Correct the private environment file, then rerun the migration service:

```bash
docker compose up --force-recreate migrate
```

Do not leave sign-up enabled as a workaround: the dashboard container and worker run with `ALLOW_SIGNUP=false`.

## Change or recover the owner account

Dashbored is single-owner by default. Change the owner email and password from the deployment host with the recovery command; do not edit the Better Auth tables manually.

The password is read from standard input, so it never appears in the command history. The command also revokes every existing owner session.

```bash
read -r -s -p "New Dashbored password: " DASHBORED_NEW_PASSWORD
printf "\n"
printf '%s' "$DASHBORED_NEW_PASSWORD" | docker compose exec -T dashboard bun run owner:reset -- --email owner@example.com --password-stdin
unset DASHBORED_NEW_PASSWORD
```

If more than one user exists, explicitly choose the current owner to avoid changing the wrong account:

```bash
printf '%s' "$DASHBORED_NEW_PASSWORD" | docker compose exec -T dashboard bun run owner:reset -- --current-email previous-owner@example.com --email owner@example.com --password-stdin
```

Run the command on the Docker host and use the same Compose project and environment file as the deployment. For a published image, upgrade to an image version that includes this command first.

## Additional users

There is no public sign-up page in v0.1. The dashboard deliberately runs with `ALLOW_SIGNUP=false` after bootstrap, because it controls private infrastructure. Do not expose a public registration form for a home-server control plane.

Multi-user access needs an explicit invitation and role model—not an environment flag alone—and is planned separately. Until then, share neither the owner password nor the dashboard outside the people permitted to operate the host.

## Origins and cookies

Authentication uses the canonical `BETTER_AUTH_URL`. Set it to exactly what a browser uses, including `https://` and any non-default port. Add that origin to `BETTER_AUTH_TRUSTED_ORIGINS`.

Examples:

```dotenv
BETTER_AUTH_URL=https://dashbored.home.arpa
BETTER_AUTH_TRUSTED_ORIGINS=https://dashbored.home.arpa
```

For more than one approved private origin, separate origins with commas. Avoid mixing an IP address, `.local`, and a hostname unless each is explicitly configured; browser cookies and origin checks treat them as separate sites.

## Passkeys

Passkeys require a secure, stable browser origin. Use HTTPS through a trusted reverse proxy, a private certificate your devices trust, or a compatible private overlay. Do not rely on a raw HTTP IP address for passkey enrollment.

If you change the hostname later, browsers treat it as a new relying party. Plan the final private hostname before enrolling passkeys.

## Session behavior

Sessions are stored in PostgreSQL, expire after 14 days, and are refreshed while active. Changing `BETTER_AUTH_SECRET` invalidates current sessions.

Never share the bootstrap password, `BETTER_AUTH_SECRET`, or browser session cookies. See [Security model](/docs/security) for the rest of the protection model.
