description: Poll responsibly, retain compact history, and keep resource use predictable.

Dashbored is designed to be quiet on a small home server. The dashboard does not continuously poll from every browser tab; the worker does the upstream work on a schedule and persists compact results for the UI.

## Polling behavior

The worker uses bounded concurrency, start-time jitter, and exponential backoff. A healthy service records a compact heartbeat rather than a full snapshot every cycle. When a service repeatedly fails, its next check is delayed instead of hammering an unavailable endpoint.

Use the default one-minute interval unless you have a concrete operational need for faster updates. Reducing it increases CPU, database writes, and upstream API traffic across every enabled service.

## Inspect the stack

```bash
docker compose ps
docker compose logs --tail=200 dashboard
docker compose logs --tail=200 worker
docker compose exec postgres pg_isready -U dashbored -d dashbored
```

A healthy web process does not guarantee that scheduled discovery and polling are running. Check `worker` separately whenever dashboards look stale.

## Web readiness

The dashboard exposes a minimal unauthenticated readiness endpoint. It reports `200` only when the web process can also query PostgreSQL; it never returns application configuration, service health, or credentials.

```bash
curl -fsS http://127.0.0.1:43821/api/health
# {"status":"ok"}
```

Compose uses this endpoint for the dashboard healthcheck. A private reverse proxy may use it for its own upstream health probe.

## Backup

Back up all three of these items:

1. The PostgreSQL database or Compose volume.
2. Your private environment file, including `BETTER_AUTH_SECRET` and `CREDENTIAL_MASTER_KEY`.
3. The repository version or image tag you deployed.

A database backup without its matching credential master key restores service records but cannot decrypt their saved credentials. A master key backup without the database does not restore configuration or history.

Example logical backup:

```bash
docker compose exec -T postgres pg_dump -U dashbored dashbored > dashbored.sql
```

Store the database dump and environment secrets separately, encrypted, and test restoration on a non-production host.

The source deployment includes a safe backup helper:

```bash
bun run backup
```

It creates a gzipped PostgreSQL dump and SHA-256 checksum in `backups/` with restrictive file permissions. It deliberately does not copy `.env`; preserve that file using your established encrypted backup system because it contains the credential master key.

For a stack started with `bun run docker:local`, use `bun run backup:local` instead.

## Restore outline

1. Stop the stack.
2. Restore the database volume or create an empty database and import a logical dump.
3. Restore the exact environment values, especially `CREDENTIAL_MASTER_KEY`.
4. Start `postgres`, run migrations, then start the rest of the stack.
5. Sign in and verify a service connection without triggering a container action.

The restore helper is intentionally explicit because it drops and recreates the Dashbored database:

```bash
bun run restore -- /absolute/path/to/dashbored.sql.gz --confirm=RESTORE
```

For a local helper-created stack, use `bun run restore:local -- /absolute/path/to/dashbored.sql.gz --confirm=RESTORE`. The script verifies a neighboring `.sha256` file when present, stops the web and worker services, restores the database, then starts the stack again.

## Upgrade

Read release notes, back up first, then rebuild and let the migration container complete:

```bash
git pull --ff-only
docker compose build
docker compose up -d
docker compose ps
```

Do not delete volumes during a normal upgrade. Inspect `migrate` logs if `dashboard` or `worker` does not start; they depend on successful migrations.

For published-image deployments, follow [Releases and upgrades](/docs/releases).

## Resource tuning

- Keep `POLL_INTERVAL_MS` at 60 seconds or longer.
- Disable or delete services you do not use.
- Prefer Portainer inventory sync rather than separate generic checks for every container.
- Keep PostgreSQL private and on local storage with regular backups.
- Avoid dashboard replicas unless you also understand the worker's advisory locking behavior; one worker performs polling at a time.
